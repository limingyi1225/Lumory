# Lumory 大规模 Bug Hunt 计划（性能 / 功能 / 使用体验）

## 0. 目标与原则

### 核心目标
- **性能**：减少启动卡顿、滚动掉帧、写入阻塞、AI 与同步链路长尾延迟。
- **功能**：提高数据一致性（CoreData/CloudKit/导入导出/回填）、降低错误恢复成本、保证 SSE 与重试语义正确。
- **使用体验**：降低「不可解释失败」比例，提升输入安全感（不丢文）、加载可感知性、错误提示可行动性。

### 执行原则
1. **先观测后修复**：每个模块先补监控点与复现脚本，再动逻辑。
2. **高风险先隔离**：与数据丢失、同步错乱、崩溃相关问题优先级最高。
3. **可回滚**：每一批修复必须可单独关闭（Feature Flag / 开关 / 分支发布）。
4. **闭环验证**：每个 bug 必须有“复现用例 + 修复验证 + 回归清单”。

---

## 1. 战役范围与优先级

### P0（必须先打）
1. **输入与数据安全**：前后台切换、崩溃恢复、CoreData save 时机导致的文本丢失。  
2. **同步一致性**：CloudKit remote change、冲突合并、重复回填、跨设备延迟与错位。  
3. **AI 流式链路稳定性**：SSE 断流、半截响应误判成功、重试幂等与 UI 状态机异常。  
4. **启动路径风险**：初始化串行阻塞、迁移/回填与主线程竞争导致 watchdog 风险。

### P1（第二批）
1. **列表与详情性能**：Home/Insight 大数据集下首帧、滚动、筛选、图表渲染。  
2. **导入导出鲁棒性**：大体积 JSON、脏数据、字段缺失、图片与音频关联完整性。  
3. **搜索质量与响应**：关键词命中、分词边界、查询延迟与 UI 卡顿。

### P2（持续治理）
1. **本地化与可访问性**：中英切换、长文案折行、Dynamic Type、VoiceOver。  
2. **观感一致性**：空状态、错误状态、进度提示与触觉反馈一致化。  
3. **后端运维韧性**：限流边界、日志可追踪性、健康探针与告警噪音治理。

## 2. 组织方式（2 周冲刺，可滚动）

### 角色分工
- **Bug Marshal（1人）**：维护看板、优先级、冻结策略、每日收敛。  
- **Client Pod（2~3人）**：iOS 性能、CoreData、CloudKit、UI 回归。  
- **Server Pod（1~2人）**：SSE、重试、限流、代理健壮性。  
- **QA/Data（1~2人）**：复现脚本、数据集构建、回归矩阵执行。

### 节奏
- D1-D2：埋点、基线采样、复现资产搭建。  
- D3-D5：P0 修复与快速回归。  
- D6-D8：P1 修复与性能专项。  
- D9-D10：P2、技术债、文档沉淀、发布复盘。

## 3. 缺陷分类标准（统一 triage 口径）

### 严重级别
- **S0**：数据不可恢复丢失 / 启动崩溃 / 无法记录。  
- **S1**：核心路径失败（保存、同步、AI 对话、导入导出）。  
- **S2**：可绕过但明显影响效率（慢、偶发失败、状态错误）。  
- **S3**：体验瑕疵（提示不清、动画抖动、视觉一致性问题）。

### 标签维度
- domain: `startup | editor | sync | ai-stream | import-export | insights | search | settings | backend`
- nature: `perf | logic | concurrency | data | ux | i18n | a11y | observability`
- reproducibility: `always | intermittent | rare`
- blast radius: `single-device | multi-device | all-users`

## 4. 模块化 Bug Hunt 清单

## 4.1 启动链路（ChronoteApp / Persistence / Migration）

### 重点风险
- 初始化期间主线程工作过多，首屏耗时抖动。
- 迁移/回填任务与 UI 生命周期竞争，造成卡顿或状态错乱。
- store 恢复路径边界条件不足。

### 检查项
1. 记录启动阶段时间线：进程启动 → Splash 出现 → Home 可交互。  
2. 分析是否有同步 I/O 或主线程重活（数据库、文件、权限串行）。  
3. 验证恢复流程：模拟数据库损坏、恢复后数据完整性与提示文案。  
4. 检查 remote change observer 注册/释放时机，避免重复监听。

### 目标指标
- 冷启动 P95 可交互时间下降 ≥ 20%。
- 启动期间主线程长任务（>100ms）数量下降 ≥ 30%。

---

## 4.2 编辑器与保存链路（Home / DiaryDetail / CoreData）

### 重点风险
- 高频输入触发过密 save，导致顿挫。
- 场景切换时 save 时机不一致导致内容回退。
- 图片/音频附件引用断裂。

### 检查项
1. 长文输入（3k~8k 字）持续 3 分钟：观察 CPU、内存、掉帧、save 频率。  
2. 切后台/杀进程/低电量模式：验证最后输入是否落盘。  
3. 附件增删改、跨天编辑、日期变更后引用完整性。  
4. 字数统计与文本实际一致性（中英文混排、emoji、换行）。

### 目标指标
- 连续输入场景平均帧率提升；明显卡顿投诉下降。
- “文本回退/丢失”类问题归零（S0 级别）。

---

## 4.3 CloudKit 同步与回填服务

### 重点风险
- Remote change 触发风暴导致重复扫描。
- Backfill 并发互斥不完善导致状态悬挂。
- 弱网下同步状态不透明，用户误判“已同步”。

### 检查项
1. 双设备 A/B 写入冲突测试：同条目并发编辑 + 离线重连。  
2. 高频 remote change（批量导入）下 backfill CPU/电量影响评估。  
3. 中断恢复测试：回填任务取消/失败后再次进入可否继续。  
4. 状态 UI 文案是否区分：同步中、失败、等待网络、权限问题。

### 目标指标
- 冲突后最终一致性成功率 > 99%。
- 回填相关耗时与失败率可观测（100% 有日志链路 ID）。

---

## 4.4 AI 流式链路（OpenAIService / NetworkRetry / server）

### 重点风险
- SSE 半截流被误判成功。
- 重试策略导致重复回答、状态机卡死、按钮不可用。
- 限流与超时时 UX 不可恢复（只能强退）。

### 检查项
1. 注入网络故障：超时、RST、分段丢包、200 后断流。  
2. 验证 DONE 语义与错误硬断的一致性。  
3. 多次连续追问（10~20 轮）下内存增长与取消能力。  
4. 429/5xx/上游超时时提示是否提供“重试/稍后再试/复制草稿”。

### 目标指标
- SSE 异常场景误判成功率降至 0。
- AI 会话可恢复率（用户可继续操作）> 99%。

---

## 4.5 Insights / Search / 图表渲染

### 重点风险
- 大数据量下聚合查询阻塞，图表首屏慢。
- 时间范围切换触发重复计算。
- 搜索联想与结果更新造成主线程抖动。

### 检查项
1. 构造 1k/5k/10k 日记数据集，测量各视图首帧与切换延迟。  
2. 识别 O(n²) 或重复 fetch；引入缓存命中统计。  
3. 图表组件分层渲染与数据预处理解耦验证。  
4. 搜索输入防抖、取消上次查询、空结果反馈质量。

### 目标指标
- Insights 首次可用时间下降 ≥ 25%。
- 搜索输入到结果 P95 < 300ms（本地数据场景）。

---

## 4.6 导入导出与迁移

### 重点风险
- 老版本 JSON 字段兼容问题。
- 大文件导入内存峰值过高导致 OOM。
- 导出后再导入不完全可逆。

### 检查项
1. 用历史样本集执行回放（含坏字段、缺字段、超长文本）。  
2. 图片/音频/主题/embedding/wordCount 全字段一致性校验。  
3. 导入中断（来电、切后台、杀进程）恢复策略验证。  
4. 导出产物跨设备可读性（时区、编码、文件命名冲突）。

### 目标指标
- 导入成功率 > 99%，失败可解释率 100%。
- round-trip（一导一回）字段一致率 > 99.9%。

---

## 4.7 设置页与可运维性

### 重点风险
- “一键重建索引”执行状态不清晰，用户误触重复执行。
- 同步诊断信息不足，客服难定位。
- Secret/配置错误时提示不友好。

### 检查项
1. 各重建入口互斥与状态同步（进行中/已完成/失败）。  
2. 诊断页输出结构化：时间戳、设备、网络、错误码、建议动作。  
3. 后端配置缺失时客户端可识别并给出可行动提示。

### 目标指标
- 用户自助恢复成功率提升（无需人工介入）。
- 设置页相关支持工单量下降。

## 5. 观测与工具化（先做，再 hunt）

### 客户端埋点（建议）
- 启动阶段耗时：`app_launch_phase_*`
- 保存链路：`entry_save_start/end/error`
- 同步链路：`cloudkit_remote_change_received`, `backfill_run_*`
- AI 链路：`sse_open/chunk/done/error/retry/cancel`
- UX：空状态曝光、错误弹窗曝光、重试点击率

### 后端埋点（建议）
- 请求级 trace id（客户端透传）。
- SSE 生命周期日志：open/chunk_count/done/destroy/error_type。
- 429、5xx、timeout 分桶。
- 上游 OpenAI 响应时间 P50/P95/P99。

### 工具栈
- iOS Instruments: Time Profiler / Allocations / Main Thread Checker / Points of Interest。
- CoreData SQLDebug + signpost。
- 网络故障注入：Network Link Conditioner / 本地代理故障脚本。
- 后端压测：k6 或 autocannon（chat/embedding 分开）。

## 7. 输出物与验收标准

### 每个 bug 必须交付
1. 复现步骤（可脚本化优先）。  
2. Root Cause（代码点 + 条件）。  
3. 修复方案（含回滚策略）。  
4. 自动化/半自动回归用例。  
5. 风险评估（影响面 + 兼容性）。

### 冲刺验收（建议）
- S0 关闭率 100%，S1 关闭率 ≥ 90%。
- 性能核心指标达到目标（启动、Insights、搜索、SSE 稳定性）。
- 关键路径（写日记、同步、AI 对话、导入导出）无 blocker。

## 9. 风险与防回归策略

- **风险1：快速修复引入新并发问题**  
  - 对策：所有并发改动必须附带取消/重入测试。
- **风险2：优化后结果不一致**  
  - 对策：性能改造前后做结果快照对比。
- **风险3：日志噪音过大影响定位**  
  - 对策：统一 error code，降级 debug 日志采样率。
- **风险4：跨端改动不同步**  
  - 对策：客户端与后端约定版本化协议与灰度窗口。

## 10. 执行建议（落地顺序）

1. 当天先建 bug board（按 P0/P1/P2 + S0/S1/S2/S3 双维度）。  
2. 当天完成最小观测集（启动、保存、SSE、同步四条主链路）。  
3. 第 2 天开始只接“可复现+可验证”的 bug 进入开发。  
4. 每晚做 30 分钟 triage：升级/降级、阻塞清理、第二天聚焦。

> - **首批自动化回归脚本清单**（xcodebuild + server 健康检查 + 故障注入）
